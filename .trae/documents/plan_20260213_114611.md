I will refactor the `ExternalCalculation` architecture to separate the "Account" entity from its "Transactions".

### 1. Database & Migration
- Rename the existing `external_calculations` table to `external_calculation_entries` (preserving transaction data).
- Create a new `external_calculations` table to act as the "Parent Account" with fields: `id`, `farm_id`, `name`, `description`, `timestamps`.
- Add `external_calculation_id` foreign key to the `external_calculation_entries` table.
- **Data Migration**: Create a default "General External Account" and assign all existing transactions to it to prevent data loss.

### 2. Model Refactoring
- **Create `ExternalCalculationEntry` Model**:
  - Move transaction-specific logic (relationships to `Account`, `JournalEntry`, casts for `amount`/`date`) from the current model to this new one.
  - Add a `belongsTo` relationship to the parent `ExternalCalculation`.
- **Update `ExternalCalculation` Model**:
  - Strip it down to be the parent entity (Name, Description).
  - Add a `hasMany` relationship to `ExternalCalculationEntry`.

### 3. Logic & Observers
- **Rename/Refactor Observer**: Convert `ExternalCalculationObserver` to `ExternalCalculationEntryObserver`.
- Ensure accounting posting logic listens to the creation of the *Entry*, not the Account.

### 4. Filament Resource Updates
- **`ExternalCalculationResource`**:
  - Update the Form to only accept `Name` and `Description`.
  - Update the Table to list External Accounts.
  - Register a new `EntriesRelationManager` to manage the transactions within the Account view.
- **`EntriesRelationManager`**:
  - Implement the form and table logic previously found in the main resource (Amount, Date, Type, etc.).
